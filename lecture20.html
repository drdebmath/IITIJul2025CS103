Introduction to Google C++ Style Guide

What is the Google C++ Style Guide?

The Google C++ Style Guide provides detailed rules for writing C++ code that is manageable, readable, and maintainable at scale.

  • Goals: Manage complexity, optimize for readers, ensure consistency
  • Scope: Style and readability conventions
  • Application: All Google open-source C++ projects

The Google C++ Style Guide is a comprehensive document that outlines the dos and don'ts of writing C++ code. It exists to manage the complexity inherent in C++ while allowing developers to use language features productively.

Background

C++ is powerful but complex. The guide helps teams write code that is:

  • Readable: Easy to understand and maintain
  • Efficient: Performs well and scales
  • Consistent: Follows uniform conventions
  • Safe: Avoids common pitfalls and bugs

The guide emphasizes that "style" encompasses far more than just source file formatting—it's about making code manageable while allowing productive use of C++ features.

Goals and Guiding Principles

Core Goals of the Style Guide

  • Style rules must pull their weight - Benefits justify the cost
  • Optimize for the reader, not the writer - Most code is read, not written
  • Be consistent - With existing code and broader C++ community
  • Avoid surprises and dangers - Prevent error-prone constructs
  • Mindful of scale - Google has 100M+ lines of C++ code

Style Rules Should Pull Their Weight

The benefit of a style rule must be large enough to justify enforcing it across thousands of engineers. Rules exist because the alternatives are worse—not because they represent some philosophical ideal.

Optimize for the Reader, Not the Writer

Codebases endure far longer than the initial implementation. Most time is spent reading and understanding code, so readability optimizations are crucial.

Be Consistent with Existing Code

Using one style consistently lets engineers focus on important issues. Consistency also enables tool automation.

Avoid Surprising or Dangerous Constructs

C++ has features that appear safe but have subtle dangers. The guide prohibits such constructs, and waivers require strong justification.

Be Mindful of Scale

With millions of lines of code and thousands of engineers, even small mistakes can become costly. Rules that seem unnecessary in small projects are justified at scale.

C++ Version and Extensions

C++ Version Guidelines

  • Target C++20 - Don't use C++23 features yet
  • No non-standard extensions - Platform-specific code ok only in system code
  • Consider portability - C++17/20 features may not be universally supported

The C++ version targeted by the guide advances aggressively over time. Currently, code should target C++20 and avoid C++23 features.

C++ Version Progression

The guide's target C++ version advances over time. Currently:

  • Target: C++20
  • Avoid: C++23 features
  • Future: Version will advance as ecosystem support matures

Non-Standard Extensions

Do not use non-standard extensions. Examples to avoid:

  • GCC's __attribute__
  • Microsoft's __declspec
  • Compiler intrinsics like __builtin_prefetch
  • Inline assembly
  • Variable-length arrays (alloca(), VLAs)

Portability wrappers using extensions are allowed, but only through designated project-wide headers.

Header Files

Header File Best Practices

  • Self-contained headers - Compile on their own
  • #define guards - Prevent multiple inclusion
  • Include what you use - Direct includes only
  • Avoid forward declarations when possible
  • Order includes - Related, C, C++, libraries, project

Correct use of header files dramatically impacts code readability, size, and performance.

Self-Contained Headers

All header files should be self-contained and end in .h.

  • Use header guards: #ifndef PROJECT_PATH_FILE_H_
  • Include all necessary headers
  • Avoid .inc files except for unusual cases

Include What You Use

Include headers directly for symbols you use, not through transitive includes.

Include Order

  1. Related header (e.g., foo.h in foo.cc)
  2. C system headers
  3. C++ standard library
  4. Other libraries
  5. Your project's headers

Forward Declarations

Avoid forward declarations where possible. They save compile time but create dependencies and risks.

// Avoid - forward declaration
class B;
void FuncInB();
ABSL_DECLARE_FLAG(flag_in_b);

Scoping

Scoping Guidelines

  • Namespaces - Place code in namespaces, unique project-based names
  • Internal linkage - Use unnamed namespaces or static for file-local code
  • Nonmember functions in namespaces - Avoid global functions
  • Local variables - Narrowest scope possible, initialize at declaration
  • Static/global variables - Avoid unless trivially destructible

Namespaces

Place code in namespaces to avoid global scope pollution. Use unique, project-based names.

namespace myproject {

void MyFunction() {
  // Code here
}

}  // namespace myproject

Internal Linkage

Use unnamed namespaces or static declarations for code not referenced outside the file.

namespace {
const int kInternalConstant = 42;
}  // namespace

Local Variables

Declare variables in the narrowest possible scope and initialize them at declaration.

// Good
int i = f();  // Declaration with initialization
f(i);         // Use immediately after

// Bad
int j;       // Declaration separate from initialization
j = g();  // Initialization later

Static and Global Variables

Objects with static storage duration are forbidden unless trivially destructible. Prefer constexpr for constants.

Classes

Class Design Guidelines

  • Work in constructors - Avoid virtual calls, ensure no-fail initialization
  • Implicit conversions - Use explicit for constructors and operators
  • Copyable/movable types - Make APIs clear about copy/move semantics
  • Structs vs classes - Structs for passive data, classes for functionality
  • Access control - Make data members private
  • Declaration order - Types, constants, factory functions, constructors, methods

Doing Work in Constructors

Constructors should avoid virtual function calls and initialization that can fail without a way to signal errors.

Implicit Conversions

Don't define implicit conversions. Use the explicit keyword for conversion operators and single-argument constructors.

class MyClass {
 public:
  explicit MyClass(int x);  // Explicit constructor
  explicit operator bool() const;  // Explicit conversion
};

Copyable and Movable Types

A class's public API must make clear whether it's copyable, move-only, or neither. Support copying/moving if meaningful.

Structs vs. Classes

Use struct only for passive objects that carry data; everything else is a class.

Inheritance

Composition is often more appropriate than inheritance. When using inheritance, make it public and prefer to avoid multiple implementation inheritance.

Operator Overloading

Overload operators judiciously. Do not use user-defined literals. Define operators only on your own types.

Functions

Function Guidelines

  • Inputs and outputs - Return values over output parameters
  • Write short functions - < 40 lines when possible
  • Function overloading - Reader should understand which overload is called
  • Default arguments - OK if default always has same value
  • Trailing return types - Use where ordinary syntax is impractical

Inputs and Outputs

Prefer using return values over output parameters. They improve readability and often provide the same or better performance.

Write Short Functions

Prefer small and focused functions. If a function exceeds about 40 lines, consider breaking it up.

Function Overloading

Use overloaded functions only if a reader can get a good idea of what's happening without knowing which overload is called.

Default Arguments

Default arguments are allowed on non-virtual functions when the default is guaranteed to always have the same value.

Trailing Return Type Syntax

Use trailing return types only where the ordinary syntax (leading return types) is impractical or much less readable.

// Prefer ordinary syntax
int func(int x);

// Use trailing when needed
auto func(int x) -> int;

Google-Specific Magic

Google-Specific Conventions

  • Ownership and smart pointers - Single, fixed owners; prefer std::unique_ptr
  • cpplint - Use cpplint.py to detect style errors
  • Exception handling - Do not use C++ exceptions
  • RTTI - Avoid run-time type information
  • Casting - Use C++-style casts, not C casts

Ownership and Smart Pointers

Prefer to have single, fixed owners for dynamically allocated objects. Prefer smart pointers for explicit ownership transfer.

// Good - clear ownership
std::unique_ptr<Foo> FooFactory();
void FooConsumer(std::unique_ptr<Foo> ptr);

cpplint

Use cpplint.py to detect style errors. It has false positives/negatives but is valuable for consistency.

Exceptions

Do not use C++ exceptions in Google code. They are allowed in third-party code and system libraries outside your control.

RTTI

Avoid using run-time type information (RTTI). Queries like typeid() and dynamic_cast<> are generally prohibited.

Casting

Use C++-style casts like static_cast(), not C-style (float)x.

Other C++ Features

Additional C++ Features

  • Rvalue references - Special cases only (move constructors)
  • Friends - Allowed within reason
  • noexcept - Specify when useful and correct
  • Streams - Use for appropriate I/O, stick to simple usage
  • Preincrement/Predecrement - Use prefix form (++i)

Rvalue References

Use rvalue references (&&) only in certain special cases: move constructors/assignments, forwarding references with std::forward.

Friends

Friends extend but do not break encapsulation. Common use: factory functions.

noexcept

Specify noexcept when it accurately reflects the function's exception behavior.

Streams

Use streams where appropriate, especially for debug logging. Avoid complex formatting and stateful operations.

Use of const

Use const whenever it makes sense in APIs. constexpr is better for true constants.

Integer and Floating-Point Types

Of built-in types, only int and double/float are used. Use exact-width types when size guarantees are needed.

Naming

Naming Conventions

  • Type names - PascalCase (MyClass)
  • Variable names - snake_case (my_variable)
  • Function names - PascalCase (MyFunction)
  • Constants - kName (kConstantName)
  • Namespaces - snake_case (my_namespace)

General Naming Rules

Give things names that describe their purpose clearly. Avoid abbreviations unless they're widely known.

When Naming Things

  • Choose descriptive names, not clever ones
  • Abbreviations: OK if listed in Wikipedia or widely known (e.g., HTML)
  • Scope affects descriptiveness - globals need more explanation than locals

Type Names

Types (classes, structs, enums) use PascalCase: starts with capital, each word capitalized.

Variable Names

Variables and data members use snake_case. Class data members have trailing underscores.

Function Names

Functions use PascalCase. Accessors/mutators may use snake_case to match their variable.

Constants

Constants start with k and use PascalCase: kConstantName.

Formatting

Code Formatting

  • Line length - Maximum 80 characters
  • Non-ASCII - Rare, use UTF-8
  • Spaces vs tabs - Only spaces, indent 2
  • Function formatting - Specific rules for declarations
  • Bracing - K&R style with spaces

Line Length

Each line of text should be at most 80 characters long.

Horizontal Whitespace

  • No trailing whitespace
  • Spaces around operators and after keywords
  • No spaces inside parentheses

Vertical Whitespace

Use blank lines sparingly. Only where they aid reading overall structure.

Function Declarations and Definitions

ReturnType ClassName::FunctionName(Type par_name1, Type par_name2) {
  // Implementation
}

Class Format

Sections: public, protected, private (each indented one space).

Exceptions to the Rules

When to Break the Rules

  • Existing code - Match local conventions when modifying
  • Windows code - Some exceptions for Windows-specific code
  • Common sense - Rules exist to serve goals, not as ends themselves

Remember: Common sense and good taste should prevail.

Existing Non-conformant Code

You may diverge from the rules when dealing with code that doesn't conform to this style guide. Stay consistent with the local conventions in that code.

Windows Code

Some rules have exceptions for Windows code:

  • Use Hungarian notation (e.g., iNum) for consistency
  • Exceptions are allowed for ATL/STL integration
  • Use #pragma once instead of standard guards

Final Words

The style guide serves the goals of managing complexity and ensuring maintainability. When in doubt, follow the rules, but remember that their purpose is to serve code quality—not the other way around.

Key takeaway: Write code that is readable, maintainable, and safe. The Google C++ Style Guide helps you achieve these goals.
